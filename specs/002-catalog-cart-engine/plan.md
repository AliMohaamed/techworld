# Implementation Plan: Catalog & Persistent Cart Engine

**Branch**: `002-catalog-cart-engine` | **Date**: 2026-03-19 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-catalog-cart-engine/spec.md`

## Summary

This feature builds the complete product catalog layer and guest shopping cart on top of the foundational schema established in Feature 001. It introduces a `categories` table with localized dual-language fields and soft-deletion, expands the `products` table with commercial presentation data (images, rich descriptions, pricing, COGS) while strictly preserving the decoupled `display_stock`/`real_stock` inventory model, implements a hybrid guest `cart_sessions` architecture using Convex-backed `sessionId` persistence, and exposes a secure webhook endpoint for automated WhatsApp payment receipt ingestion with a manual dashboard fallback.

## Technical Context

**Language/Version**: TypeScript / Node.js  
**Primary Dependencies**: Convex Serverless Functions, Next.js App Router, Convex File Storage  
**Storage**: Convex Database (real-time, transactional), Convex File Storage (product images, receipt media)  
**Testing**: Convex built-in testing (vitest), manual Convex dashboard validation  
**Target Platform**: Convex Backend + Next.js Frontend (App Router)  
**Project Type**: Cloud Database & Backend API + Web Application  
**Performance Goals**: Storefront product queries < 100ms; cart validation < 50ms; webhook ingestion < 200ms  
**Constraints**: Absolute Server Authority; financial fields stripped server-side; zero hardcoded roles; categories enforce parent-child integrity; cart does NOT reserve stock; webhook must be idempotent  
**Scale/Scope**: 4 new Convex tables (categories, expanded products, cart_sessions, webhook_receipts), 3 new mutation files, 1 Next.js API route for webhook endpoint

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- [x] **Server Authority Gate**: All product queries, cart validation, and webhook ingestion happen exclusively in Convex mutations/queries. The Next.js frontend only renders data and POSTs sessionIds — it holds zero pricing or inventory authority.
- [x] **Zero Floor & Concurrency Gate**: Cart additions do NOT touch `real_stock` or `display_stock` at all — no reservation. Stock deductions remain exclusively at the `CONFIRMED` FSM transition from Feature 001. This feature does not change that constraint.
- [x] **RBAC & Financial Opacity Gate**: Product queries serving the public storefront strip `cogs` and margin data server-side. Dashboard mutations such as `createCategory`, `publishProduct`, `attachReceipt` are all gated by `requirePermission(ctx, 'MANAGE_CATEGORIES')`, `requirePermission(ctx, 'MANAGE_PRODUCTS')`, etc. `VIEW_FINANCIALS` is mandatory to see COGS.
- [x] **Audit Gate**: Every category `isActive` toggle, product publication state change, and webhook receipt attachment generates an immutable `audit_logs` entry via `logAudit`.

## Implementation Phases

### Phase 1: Schema Expansion (Blocking — All Features Depend On This)

Expand `convex/schema.ts` with the four new/modified tables:
- `categories` table (new)
- `products` expansion (add fields to existing)
- `cart_sessions` table (new)
- `webhook_receipts` table (new)

### Phase 2: Category Engine

Implement `convex/categories.ts`:
- `createCategory` — requires `MANAGE_CATEGORIES` permission
- `updateCategory` — requires `MANAGE_CATEGORIES` permission
- `toggleCategoryActive` — requires `MANAGE_CATEGORIES` permission; logs audit entry
- `listActiveCategories` — public query, strips no sensitive data
- `getCategoryById` — public query for storefront category pages

### Phase 3: Product Catalog Engine

Expand `convex/products.ts`:
- `createProduct` — requires `MANAGE_PRODUCTS`; validates `categoryId` exists and is active
- `publishProduct` — requires `MANAGE_PRODUCTS`; validates parent category `isActive = true`; logs audit entry
- `getProduct` — public query; strips `cogs` if caller lacks `VIEW_FINANCIALS`
- `listProductsByCategory` — public query; filters to published products whose category is active
- Extend `Permission` type in `convex/lib/rbac.ts` with new flags

### Phase 4: Guest Cart Engine

Implement `convex/cart.ts`:
- `addToCart(sessionId, productId, quantity)` — validates `display_stock >= quantity`, upserts cart item
- `removeFromCart(sessionId, productId)` — removes item from session cart
- `validateCart(sessionId)` — server-side validation of all items against live `display_stock`; returns per-item pass/fail
- `getCart(sessionId)` — returns current cart contents
- `clearCart(sessionId)` — empties the cart (called post-order creation)

### Phase 5: Webhook & Receipt Ingestion

Implement the webhook pipeline:
- `app/api/webhook/whatsapp/route.ts` — Next.js API Route: verify HMAC/shared-secret signature; extract raw payload; call `internal.webhooks.processWebhookReceipt` via Convex HTTP action
- `convex/webhooks.ts` — `processWebhookReceipt` internal mutation: parse Order ID short-code from text, find matching order, attach media reference, log audit, handle unmatched payloads gracefully
- `convex/webhooks.ts` — `attachReceiptManually` mutation: requires `VERIFY_PAYMENTS`; attaches agent-uploaded receipt to a pending order; logs audit entry

### Phase 6: RBAC Extension

Extend `convex/lib/rbac.ts` `Permission` type:
- Add `MANAGE_CATEGORIES`
- Add `MANAGE_PRODUCTS`
- Add `MANAGE_DISPLAY_STOCK`
- Add `ADJUST_REAL_STOCK`

Update `convex/schema.ts` `users` permissions union to include new flags.
